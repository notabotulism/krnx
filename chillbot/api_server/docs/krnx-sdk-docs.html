Quickstart

Get KRNX running in under 30 seconds. The only external dependency is Redis.

1. Start Redis

# Using Docker
docker run -d --name krnx-redis -p 6379:6379 redis:alpine

# Or install locally
brew install redis && redis-server

2. Initialize the Kernel

from krnx import KRNXClient

# Local mode (embedded kernel)
client = KRNXClient(
    data_path="./krnx-data",
    redis_host="localhost"
)

# Or remote mode (HTTP API)
client = KRNXClient(base_url="http://localhost:6380")

3. Store and Recall Memories

# Store a memory
event_id = client.remember(
    content={"query": "What's our Q4 budget?", "response": "$2.5M"},
    workspace="acme-corp",
    user="alice",
    channel="finance"
)

# Recall recent memories
memories = client.recall(
    workspace="acme-corp",
    user="alice",
    limit=10
)

# Time-travel: What did we know yesterday?
yesterday = time.time() - 86400
history = client.replay(
    workspace="acme-corp",
    timestamp=yesterday
)
✓
That's it! You now have a temporal memory kernel that persists across sessions, supports time-travel queries, and automatically handles hash-chain provenance.

Architecture

KRNX is built as a three-layer stack: API, Fabric, and Kernel. Each layer has a single responsibility.

API Server
FastAPI · REST · WebSocket
↓
Memory Fabric
Context · Identity · Enrichment
↓
Kernel
Events · Replay · Provenance
↓
STM (Redis)
LTM (SQLite)
Analytics (DuckDB)
⚡

STM — Hot Tier

Redis-backed short-term memory. Sub-millisecond writes, 24-hour retention, consumer groups for multi-agent coordination.

💾

LTM — Warm Tier

SQLite with WAL mode. 30-day queryable storage, batch writes via executemany(), automatic archival to cold tier.

🔍

Enrichment Pipeline

Deterministic metadata enrichment. Salience scoring, relation detection, entity extraction—all without LLM calls.

Constitution

KRNX operates under a strict constitutional philosophy. These principles are inviolable.

📜
The Constitution ensures KRNX remains pure infrastructure—dumb, reliable, composable. We don't compete with memory frameworks; we empower them.
Principle Meaning
Purity The kernel remains simple, durable, minimal. No magic. No guessing. No hidden state.
Modularity Every component stands alone. The stack composes. Nothing is mandatory.

Python SDK

The SDK provides a unified interface for both embedded (local) and remote (HTTP) deployments.

KRNXClient

The primary interface for interacting with KRNX. Supports both local and remote modes.

from krnx import KRNXClient

# Local mode — embedded kernel, direct Redis/SQLite access
client = KRNXClient(
    data_path="./krnx-data",      # SQLite storage path
    redis_host="localhost",       # Redis host
    redis_port=6379,              # Redis port
    default_workspace="default", # Default workspace ID
    enable_hash_chain=True,      # Cryptographic provenance
)

# Remote mode — HTTP API, can run anywhere
client = KRNXClient(
    base_url="http://localhost:6380",
    api_key="sk-...",           # Optional API key
    timeout=10.0,               # Request timeout
)

# Context manager for automatic cleanup
with KRNXClient(data_path="./data") as client:
    client.remember("Hello, KRNX!")
Parameter Type Description
data_path str optional Path for SQLite storage. Triggers local mode.
redis_host str optional Redis host for local mode. Default: "localhost"
base_url str optional HTTP URL for remote mode. Triggers remote mode.
default_workspace str optional Default workspace for operations. Default: "default"
enable_hash_chain bool optional Enable SHA-256 hash-chain provenance. Default: False

remember()

Store a memory. The fundamental write operation.

event_id = client.remember(
    content={
        "query": "Explain quantum computing",
        "response": "Quantum computing uses qubits...",
        "model": "claude-sonnet-4",
        "tokens": 1250
    },
    workspace="physics-tutor",
    user="student_42",
    session="session_abc123",
    channel="chat",
    metadata={"topic": "quantum", "difficulty": "advanced"},
    ttl_seconds=86400  # Expire after 24 hours
)

print(f"Stored: {event_id}")
# → Stored: evt_a1b2c3d4e5f6g7h8
Parameter Type Description
content str | dict required Memory content. Strings are wrapped as {"text": "..."}
workspace str optional Workspace ID. Default: default_workspace
user str optional User ID. Default: "default"
channel str optional Channel for filtering (e.g., "chat", "code")
ttl_seconds int optional Time-to-live. None = never expires.
💡
remember() is non-blocking. Events are written to STM (Redis) immediately, then asynchronously persisted to LTM (SQLite) by the background worker.

recall()

Query memories. Supports time-range filtering and channel scoping.

# Get recent memories
memories = client.recall(
    workspace="physics-tutor",
    user="student_42",
    limit=20
)

# Filter by time range
one_hour_ago = time.time() - 3600
recent = client.recall(
    workspace="physics-tutor",
    start_time=one_hour_ago,
    end_time=time.time()
)

# Filter by channel
code_memories = client.recall(
    workspace="physics-tutor",
    channel="code"
)

Returns: List[Dict] — Each memory as a dictionary with event_id, content, timestamp, metadata, etc.

replay()

Differentiator

Time-travel query. Reconstruct state at any point in history. This is what RAG cannot do.

# What did we know at this exact moment?
point_in_time = 1704067200.0  # Jan 1, 2024 00:00:00 UTC

history = client.replay(
    workspace="trading-bot",
    timestamp=point_in_time,
    user="algo_001"
)

# Events are returned in chronological order
for event in history:
    print(f"{event['timestamp']}: {event['content']}")
🔮
Perfect for auditing and debugging. Replay exactly what an agent knew when it made a decision. No more "the model hallucinated"—you can prove what information was available.

Multi-Agent Coordination

KRNX uses Redis Streams consumer groups for coordinating multiple agents on shared memory.

# Create agent groups
client.create_agent_group(
    workspace="project-alpha",
    group="coders"
)
client.create_agent_group(
    workspace="project-alpha",
    group="testers"
)

# Agents consume events from their group
events = client.consume(
    workspace="project-alpha",
    group="coders",
    agent_id="coder_001",
    count=10,
    block_ms=5000  # Wait up to 5s for new events
)

# Process and acknowledge
for event in events:
    process(event)
    client.ack(
        workspace="project-alpha",
        group="coders",
        message_id=event['message_id']
    )

Enrichment Pipeline

KRNX's enrichment layer transforms raw events into intelligent memory through deterministic metadata computation. No LLMs. No hallucination. Just composable, transparent scoring.

✓ Constitution-Safe: Pure, deterministic, no side effects

Overview

The enrichment pipeline runs 5 stages in order:

  1. Temporal — Episode threading, time gaps
  2. Salience — Importance scoring (semantic + recency + frequency + structural)
  3. Relations — Detect supersedes, contradicts, duplicates, expands_on
  4. Entities — Extract @mentions, projects, files, concepts
  5. Retention — Classify as ephemeral, durable, consolidation_candidate
from enrichment import MetadataEnricher, EnrichmentContext

enricher = MetadataEnricher()

context = EnrichmentContext(
    previous_event=last_event,
    workspace_events=recent_events,
    similarity_scores={"evt_123": 0.87, "evt_456": 0.72}
)

metadata = enricher.enrich(event, context)
event.metadata.update(metadata.to_dict())

Salience Scoring

Salience determines which memories surface during recall. Higher salience = more likely to be retrieved.

Components

ComponentWeightDescription
semantic 40% Similarity to recent queries
recency 30% Exponential decay (24h half-life)
frequency 15% Logarithmic scaling of access count
structural 15% Position-based: episode boundaries, corrections, density 
from enrichment import SalienceEngine, SalienceMethod

engine = SalienceEngine()

result = engine.compute(
    event_id="evt_123",
    timestamp=event.timestamp,
    access_count=5,
    avg_similarity=0.82,
    structural_score=0.6,
    method=SalienceMethod.COMPOSITE
)

print(result.to_spec_dict())
# {
#   "semantic": 0.82,
#   "recency": 0.65,
#   "frequency": 0.28,
#   "structural": 0.60,
#   "final": 0.67
# }

Relation Detection

The core insight: embedding similarity alone fails critical memory cases. Two events can be highly similar but have opposite meanings ("approved" vs "not approved").

KRNX uses multi-signal feature extraction to detect:

RelationConditionsExample
duplicates sim ≥ 0.95, no contradiction Same message sent twice
supersedes sim ≥ 0.70, contradiction, newer "Budget is $75K" supersedes "Budget is $50K"
contradicts sim ≥ 0.70, contradiction, different actor Alice: "approved" vs Bob: "rejected"
expands_on sim 0.50-0.90, entity overlap Follow-up with more details
replies_to Temporal proximity, same episode Direct response in conversation

Contradiction Signals

These signals fire when events conflict:

  • negation_mismatch — "is approved" vs "is not approved"
  • numeric_mismatch — "$50K" vs "$75K"
  • temporal_mismatch — "meeting today" vs "meeting tomorrow"
  • antonym_detected — "enabled" vs "disabled"
  • cancellation_detected — "actually", "scratch that", "correction"
⚠️
strict_contradiction fires when 2+ signals are present. This indicates high-confidence conflict and triggers special handling (always retained, boosted salience).

Retention Classification

KRNX classifies events using a drift × salience matrix. Drift measures how much context has changed since the event.

Low Salience High Salience
High Drift ephemeral consolidation_candidate
Low Drift merge_candidate durable

Override: Events with strict_contradiction are always classified as durable.

Kernel Internals

For those who want to understand (or extend) the internals.

Storage Tiers

STM (Redis)

  • 24-hour retention
  • <1ms writes via pipeline
  • Consumer groups for multi-agent
  • Connection pooling (200 max)

LTM Warm (SQLite)

  • 30-day queryable storage
  • WAL mode, 64MB cache
  • executemany() for 10-50x faster batch writes
  • Disabled autocheckpoint in high-throughput mode

LTM Cold (Archive)

  • Compressed (zlib level 6)
  • Forever retention
  • Automatic archival from warm tier

Backpressure System

KRNX protects itself from overload with hysteresis-based backpressure.

MetricEngage ThresholdRecovery Threshold
queue_depth 50,000 events 37,500 events (75%)
lag_seconds 30 seconds 22.5 seconds (75%)

When backpressure engages, writes throw BackpressureError. The 25% hysteresis gap prevents oscillation.

from krnx import BackpressureError

try:
    client.remember(content)
except BackpressureError:
    # System is overloaded, back off and retry
    time.sleep(1)
    client.remember(content)

Crash Recovery

KRNX handles worker crashes gracefully via Redis Streams pending message reclamation.

from krnx.recovery import run_recovery
from krnx.connection_pool import get_redis_client

# Run at application startup
result = run_recovery(
    redis_client=get_redis_client(),
    stream="krnx:ltm:queue",
    group="krnx-ltm-workers",
    idle_timeout_ms=300000  # 5 minutes
)

print(f"Reclaimed {result['messages_reclaimed']} messages from {len(result['dead_consumers'])} dead consumers")
